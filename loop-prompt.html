============================================================
Build in this sequence, committing after each numbered step:

BUSINESS RULES (implement AND unit-test every one)
 R1. Product SKU is unique            -> 409 on duplicate
 R2. Customer email is unique         -> 409 on duplicate
 R3. Product quantity never negative  -> validation + DB check
 R4. Order blocked if stock insufficient -> 400, no partial write
 R5. Creating an order auto-reduces stock (transactional)
 R6. Total amount computed by backend (client total ignored)
 R7. Deleting an order restores stock (transactional)
 R8. All request data validated (Pydantic) -> 422 on bad input
 R9. Correct HTTP codes everywhere: 200/201/204 + 400/404/409/422
     and consistent JSON errors {"detail": "..."}

PHASE 2 — REVIEW / FIX LOOP  (repeat until ZERO bugs)
============================================================
Treat this as a loop. Each iteration:

 STEP A — REVIEW (be adversarial, hunt for bugs; do not confirm success):
   • Run backend tests:        cd backend && pytest -q
   • Run/boot the stack:       docker compose up --build  (or run backend+frontend locally)
   • Exercise EVERY endpoint and the full UI flow:
       - Products CRUD incl. duplicate SKU (expect 409)
       - Customers CRUD incl. duplicate email (expect 409)
       - Create order: success path reduces stock + correct total
       - Create order: insufficient stock -> 400, no stock change
       - Delete order: stock restored
       - Delete product/customer that is referenced -> 409
       - Dashboard numbers + low-stock list are correct
       - Frontend: responsive layout, form validation, success/error toasts,
         loading + empty states, no console errors, no broken requests (CORS)
   • Check: HTTP codes, edge cases (qty 0/negative, missing fields, bad IDs,
     huge quantities), transaction atomicity (no partial writes), and that
     no credentials are hardcoded.
   • Write down EVERY defect found as a concrete list (file:line + symptom).

 STEP B — DECISION:
   • If the defect list is EMPTY and all tests pass and all flows work ->
     EXIT THE LOOP, go to PHASE 3.
   • Otherwise continue to STEP C.

 STEP C — FIX:
   • Fix the highest-impact bug(s). Keep changes minimal and correct.
   • Re-run the relevant test; add a regression test if it was missing.
   • Commit via ./commit-as-rrr.sh "fix: <what>".
   • Return to STEP A. (Do NOT declare done after a fix without re-reviewing.)

Never fake green: never hardcode expected values or special-case logic just
to pass a test. Fix the real cause. If a check fails, report the real output.
